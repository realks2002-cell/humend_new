  2. 회원이 아닌 제3의 인물을 추가해서 저장할 때

  결과: 시나리오 1과 동일 - 해당 행은 무시됨

  - DB 매칭 기준이 work_records → members JOIN이므로, members 테이블에 없는 사람은 애초에 work_records도 없음
  - 역시 174~186번 줄에서 매칭 실패 → "매칭되는 근무기록 없음" 에러
  - DB에 아무것도 저장되지 않음 (payments INSERT 자체가 안 됨)
  - 보안상 안전: service role key를 쓰더라도 존재하지 않는 work_record_id가 없으므로 INSERT 불가

  3. 시트의 상태를 "대기"에서 "지급"으로 바꾸지 않고 저장할 때

  결과: 시트의 상태값 그대로 저장됨

  212번 줄을 보면:

  status: row["상태"] || "확정",

  - 시트의 "상태" 컬럼 값을 그대로 사용합니다
  - "대기"를 그대로 두면 → payments 테이블에 **status = "대기"**로 저장됨
  - 빈값이면 → fallback으로 "확정"이 됨
  - 주의: export 시 89번 줄에서 r.status (work_record의 상태)를 내보내는데, 서명 완료된 건은 보통 "확정" 상태이므로 기본적으로 "확정"이 시트에 표시됨

  즉 관리자가 의도적으로 상태를 변경하지 않아도, export된 원래 상태값이 그대로 저장되므로 실질적인 문제는 없습니다. 다만 상태가 "대기"인 채로 payments에 저장되면, 지급 처리 단계에서 해당 건의 표시가 달라질
   수 있습니다.
